Memory and Retrieval

The memory stream is the central data structure of a generative agent: an append-only, timestamped log of natural-language records. This page documents the types that make it up and the retrieve function that selects from it. The Memory Stream guide explains the retrieval scoring in narrative form and works it through an example.

Memory entries

A MemoryEntry is one record in a stream — an observation, a reflection, or a plan, as enumerated by MEMORY_KINDS.

GABM.MemoryEntry — Type
MemoryEntry(content; kind, importance, created, last_accessed, embedding, citations)

A single record in a MemoryStream.

Fields

  • content::String: the memory itself, in natural language.
  • kind::Symbol: one of MEMORY_KINDS.
  • importance::Float64: poignancy on the Park et al. 1–10 scale, where 1 is mundane and 10 is profound. See rate_importance.
  • created::Int: the simulation tick at which the memory was formed.
  • last_accessed::Int: the tick at which the memory was most recently returned by retrieve; recency decay is measured from this value, so retrieving a memory rejuvenates it.
  • embedding::Union{Nothing,Vector{Float64}}: an optional dense vector used for relevance scoring.
  • citations::Vector{Int}: indices of the memories a reflection was derived from; empty for observations and plans.
source
GABM.MEMORY_KINDS — Constant
MEMORY_KINDS

The recognised values of the kind field of a MemoryEntry:

  • :observation — something the agent perceived or did.
  • :reflection — a higher-level inference synthesised by reflect!.
  • :plan — an intention recorded by plan!.

The distinction is the one Park et al. (2023) draw between the raw perceptual stream and the abstractions an agent builds on top of it.

The memory stream

A MemoryStream holds an agent's entries together with the parameters — a recency decay and three retrieval weights — that govern how they are recalled.

GABM.MemoryStream — Type
MemoryStream(; decay = 0.995, weights = (1.0, 1.0, 1.0))

An append-only, time-ordered log of an agent's MemoryEntry records.

Keyword arguments

  • decay::Float64: the per-tick base of the exponential recency score. With the default 0.995, a memory not accessed for 100 ticks keeps about 60% of its recency weight; a smaller value forgets faster.
  • weights::NTuple{3,Float64}: the (recency, importance, relevance) coefficients combined by retrieve. Raising one coordinate makes retrieval lean more heavily on that signal.

A fresh stream is empty; add to it with remember! and query it with retrieve.

source
GABM.remember! — Function
remember!(stream, content; kind, importance, tick, embedding, citations) -> Int

Append a memory to stream and return its index.

content is the natural-language record; the keyword arguments map onto the fields of MemoryEntry. tick is the current simulation time and is recorded as both the creation and the initial last-accessed time.

Most models do not call remember! directly: observe!, reflect!, and plan! call it with the appropriate kind.

Retrieval

retrieve is the function of Park et al. (2023): it scores every candidate memory on recency, importance, and relevance, and returns the highest-scoring few. embed_memories! populates the embeddings that let the relevance term use cosine similarity rather than the lexical fallback.

GABM.retrieve — Function
retrieve(stream, query; tick, n = 5, kinds = nothing, llm = nothing,
         weights = stream.weights) -> Vector{MemoryEntry}

Return the n memories from stream most worth surfacing for query.

This is the retrieval function of Park et al. (2023). Each candidate memory is scored on three components, each min-max normalised across the candidate set so the components are comparable:

  • Recency — decay ^ (tick - last_accessed), an exponential decay that favours memories formed or used recently (see MemoryStream).
  • Importance — the memory's stored poignancy, on the 1–10 scale.
  • Relevance — similarity to query: cosine similarity of embeddings when both the query and the memory have them, and Jaccard token overlap otherwise.

The three normalised components are combined by weights and the top n entries are returned. Retrieving a memory updates its last_accessed tick to tick, so attention is self-reinforcing: a memory used now is easier to retrieve next time.

Keyword arguments

  • tick: the current simulation time (required).
  • n: how many memories to return.
  • kinds: if given, an iterable of MEMORY_KINDS to restrict the search to (e.g. kinds = (:reflection,)).
  • llm: a backend used to embed the query when embedding-based relevance is wanted; pass the same backend that produced the memory embeddings.
  • weights: override the stream's (recency, importance, relevance) weights for this query only.
source
GABM.embed_memories! — Function
embed_memories!(stream, llm) -> MemoryStream

Compute and cache an embedding for every entry in stream that lacks one, using embed on the backend llm.

Calling this once after a batch of observations lets subsequent retrieve calls score relevance by cosine similarity instead of the lexical fallback. It is a no-op — and raises nothing — if llm does not support embeddings.
